# PlaylistMaker 🎵

Android‑приложение для поиска музыки в iTunes, прослушивания превью треков, создания плейлистов и их обмена.

Проект разработан в рамках 15‑месячного курса «Android‑разработчик» (Яндекс Практикум). Демонстрирует гибридный UI‑подход (Jetpack Compose + XML), единую навигацию через NavGraph, работу с сетью и реляционной БД, а также современные практики сборки (KSP, Firebase BOM).

## Стек технологий

- **Язык**: Kotlin (1.9+)
- **UI**:
    - Jetpack Compose (Material 3) — экраны «Поиск», «Настройки», «Медиа» (с вкладками «Избранное» и «Плейлисты» на `PrimaryTabRow`)
    - XML (ConstraintLayout, CoordinatorLayout) — остальные экраны
    - Интеграция: Composable‑функции запускаются напрямую из фрагментов через `setContent` (без ComposeView)
- **Навигация**: Jetpack Navigation Component (NavGraph) — единый граф для Compose‑ и XML‑экранов
- **Архитектура**: MVVM, чёткое разделение слоёв (Domain/Data/Presentation)
- **Сеть**: Retrofit + OkHttp, Gson, обработка ошибок и таймаутов
- **Хранение данных**:
    - Room (v2.5, KSP) — избранные треки, плейлисты
    - SharedPreferences — простые настройки и флаги
- **DI**: Koin
- **Асинхронность**: Kotlin Coroutines, Flow, `collectAsStateWithLifecycle`
- **Изображения**: Glide (XML‑экраны), Coil Compose (Compose‑экраны)
- **Аналитика**: Firebase Analytics (через Firebase BOM)
- **Сборка и инструменты**: Gradle, Android Studio Giraffe/Hedgehog, KSP (для компиляции Room)
- **Min SDK**: 29, **Compile SDK / Target SDK**: 36

## Что умеет приложение

- Поиск треков по запросу через iTunes API с отображением результатов в `LazyColumn` (Compose)
- Прослушивание превью трека
- Создание и редактирование плейлистов, добавление треков в избранное
- Экран «Медиа» с переключением между вкладками «Избранное» и «Плейлисты» через `PrimaryTabRow`
- Настройка приложения (экран настроек на Compose)
- Обмен плейлистами (локальный экспорт/базовый механизм)

## Особенности реализации UI и навигации

- **Экран «Поиск»** (Compose): декларативная вёрстка, работа со списками, оптимизация рекомпозиций через `key`, обработка состояний загрузки/ошибок.
- **Экран «Настройки»** (Compose): чистые Composable‑компоненты, привязка к состоянию ViewModel, корректная работа с жизненным циклом.
- **Экран «Медиа»** (Compose): контейнер с `PrimaryTabRow`, переключающий между «Избранное» и «Плейлисты». Синхронизация состояния табов и контента, минимизация лишних рекомпозиций.
- **Остальные экраны** — на XML (ConstraintLayout и др.): классические View‑компоненты, стабильная верстка.
- **Интеграция Compose без ComposeView**: прямой вызов `setContent` в фрагменте — современный подход, упрощающий работу с жизненным циклом и состоянием.
- **Единая навигация**: NavGraph связывает Compose‑ и XML‑экраны, обеспечивая стабильную передачу аргументов и сохранение состояния.

## Ключевые инженерные решения и задачи, которые я решил

- **Использование KSP для Room**: настроил компиляцию DAO и сущностей через KSP (вместо устаревшего kapt), что ускоряет сборку и соответствует современным практикам Android‑разработки.
- **Централизованное управление версиями Firebase**: подключил Firebase через BOM, чтобы избежать рассинхронизации версий между компонентами.
- **Гибридная UI‑архитектура**: реализовал корректную интеграцию Compose‑экранов в проект на XML без использования ComposeView, сохранив чистоту архитектуры.
- **Оптимизация UI и рекомпозиций**: в Compose‑части настроил стабильные ключи (`key`) в `LazyColumn`, чтобы избежать лишних рекомпозиций при обновлении списка.
- **Работа с состояниями**: использовал `collectAsStateWithLifecycle` для безопасного получения состояния из ViewModel с учётом жизненного цикла экрана.
- **Обработка ошибок сети и таймаутов**: централизованная обработка ошибок API, понятные пользователю сообщения, настройка таймаутов в OkHttp.
- **Безопасность секретов**: API‑ключи и чувствительные параметры не хранятся в репозитории; конфигурация через `local.properties` или механизмы сборки.

> **Пример проблемы, которую я решал**: при повороте экрана `LaunchedEffect` перезапускался, из‑за чего происходил лишний запрос к API. Решение — стабилизация входных параметров, проверка на дублирование запроса в ViewModel и аккуратная работа с состоянием экрана.
>
> **Ещё один кейс**: была проблема с отображением иконки (красная иконка становилась чёрной при запуске). Причина — несоответствие цветовых режимов/тем; решение — явное указание цвета ресурса и проверка конфигурации темы. Это помогло стабилизировать отображение UI на разных устройствах.
